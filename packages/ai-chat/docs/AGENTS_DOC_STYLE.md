# AGENTS_DOC_STYLE.md — structure for `@carbon/ai-chat` docs

Load this when authoring **net-new** pages in [packages/ai-chat/docs/](.). Routine edits don't need it. For voice and word choice, see [AGENTS_TONE.md](../../../AGENTS_TONE.md). For the JSDoc that becomes the API reference, see [../src/types/AGENTS.md](../src/types/AGENTS.md).

## What these pages are

These are **conceptual and task guides** — "how do I stream a message," "how do I theme the chat." They are not the API reference.

The API reference (every prop, method, event, and type) is **generated by TypeDoc from JSDoc** on the symbols exported from [../src/aiChatEntry.tsx](../src/aiChatEntry.tsx) and [../src/serverEntry.ts](../src/serverEntry.ts). You never hand-write a page that lists a type's properties — you write the JSDoc ([../src/types/AGENTS.md](../src/types/AGENTS.md)) and link to the generated page with `{@link}`. A guide's job is to connect tasks to those generated pages.

## Page anatomy

Each page is one markdown file with YAML frontmatter. TypeDoc renders `title` as the page heading, so **don't add an `#` H1** — start the body at `##`.

```markdown
---
title: Updating messages with upsertMessage
---

## Overview

State the task in one or two sentences. Lead with what the reader does.

## <Task sections>

...

## Related

- [Related guide](./OtherPage.md) — why you'd go there.
```

Then register the file in `projectDocuments` in [../typedoc.json](../typedoc.json) at the sidebar position you want — **a page that isn't registered never renders.** Match `categoryOrder` from the same config when you group examples.

Use `##` for sections and `###` sparingly for subsections. Open with `## Overview` and close with `## Related`.

## One page, one job

Keep each file single-purpose. If a page needs to explain a second, separable thing, split it into a child page and link to it rather than growing one long page.

Nest children with the `children:` frontmatter key — the parent becomes a hub:

```markdown
---
title: UI customization
children:
  - ./Theming.md
  - ./Layout.md
---
```

See [Customization.md](./Customization.md) for the hub-and-children pattern. **Litmus test:** if your overview paragraph needs an "and also," you have two pages.

## Ground the reader first

If a section assumes knowledge the reader might not have, give it to them — or link to it — before you dive in. Don't strand someone mid-page.

For example, slotted content assumes you can reach {@link ChatInstance}; a page that uses it should link to how you get the instance before showing the slot code. One sentence and a link is enough.

## Cross-link aggressively

Linking is how a reader navigates and how the search index and MCP server connect concepts. Over-link rather than under-link.

- **Default to `{@link SymbolName}`** for any exported type, method, or event — e.g. {@link ChatInstanceMessaging.upsertMessage}. It renders as a clickable jump, survives renames, feeds the MCP/search index, and is **build-validated**: TypeDoc runs with `validation.invalidLink` and treats warnings as errors, so a broken `{@link}` fails the build.
- **Use relative Markdown links** (`./MessageFormat.md`) between guides.
- **Use full URLs** for external references.
- **Never link into `src/`.** Source paths are internal and the site can't resolve them.

## Tables — last resort

Reach for a `{@link}`, not a table. A hand-maintained table of props, parameters, or return values duplicates TypeDoc and rots the moment the type changes.

Build a table **only when there's no exported symbol to link** — and when you hit that case, ask why. Usually it means the type should be exported and documented; raise that. The legitimate exceptions are surfaces that aren't TypeScript symbols at all:

- CSS custom-property tokens (`--cds-aichat-*`), as in [Layout.md](./Layout.md).
- String-pack keys for localization.

When you do use a table: keep cells short, code-format values (`` `string` ``), and use `-` for a required field with no default.

## Code blocks

Tag the language so the Carbon theme highlights it (` ```tsx `, ` ```ts `, ` ```html `, ` ```bash `, ` ```css `). Keep every snippet copy-pasteable and use **real exported types** — don't invent config shapes.

```tsx
import { ChatContainer } from "@carbon/ai-chat";

function App() {
  return <ChatContainer header={{ title: "My Assistant" }} />;
}
```

All examples must be tested. Follow the validation checklist in [AGENTS.md](AGENTS.md) → _Code examples in docs_.

## Admonitions

**Always blockquote a callout.** Every `Note`, `Tip`, or `Warning` must start with `> ` — never plain bold text on its own line. A callout that isn't blockquoted renders as ordinary paragraph text and loses its visual weight.

```markdown
> **Note**: The AI theme is on by default.

> **Tip**: Return the same element to preserve component identity across updates.
```

## Starter skeleton

Copy this into a new file to begin:

```markdown
---
title: <Sentence-case task title>
---

## Overview

<One or two sentences: what the reader accomplishes here. Lead with the task.>

## <First step or concept>

<Prose + a tested code block. {@link} every API you mention.>

## Related

- [<Related guide>](./Related.md) — <why you'd go there>.
```

## Before you publish

- [ ] `title` set in frontmatter; page registered in `projectDocuments` ([../typedoc.json](../typedoc.json)).
- [ ] Overview leads with the task.
- [ ] Every API mention is a `{@link}`, not a backtick or a table.
- [ ] Prerequisites are covered or linked before they're needed.
- [ ] No table that a `{@link}` could replace.
- [ ] Code blocks are language-tagged, tested, and use real exported types.
- [ ] Every `Note`/`Tip` callout is a blockquote (`> **Note**: …`).
- [ ] Single shell commands use inline backticks, not a fenced block.
- [ ] Voice follows [AGENTS_TONE.md](../../../AGENTS_TONE.md).
- [ ] `npm run build` passes (catches broken `{@link}`).
